# Steam 데이터 기반 게임 분석 서비스

## 프로젝트 개요

Steam 게임 데이터를 수집·분석하여 두 가지 서비스를 제공하는 웹 애플리케이션

- **소비자용**: 게임 구매 의사결정을 돕는 분석 정보
- **개발자/퍼블리셔용**: 게임 성과 KPI 대시보드

---

## 서비스 구성

Streamlit 대시보드 — 단일 게임 검색 후 3탭으로 분석 제공

### Main 탭 — 종합 요약
- 게임 기본 정보 (이미지, 장르, 메타크리틱, 카테고리 배지)
- 리뷰 요약 (총 리뷰 수, 긍정률 게이지, 통합평가)
- 카테고리 레이더차트 (긍정 7축 / 부정 7축)
- 패치노트/뉴스 목록 + 기준일(anchor) 설정
- 현재 가격 + 평균 동시접속자 카드
- 동시접속자 일별 라인차트
- 할인 내역 테이블

### Detail 탭 — 리뷰 & 플레이타임 분석
- KPI 카드 (총 리뷰, 통합평가, 평균 플레이타임, 긍정 키워드 수)
- 일별 플레이타임 추이 (추천/비추천 분리 라인차트)
- 일별 리뷰수 스택바차트
- 언어 비중 파이차트 (추천/비추천 분리)
- 키워드 빈도 TOP 20 (추천/비추천 분리)
- 리뷰 목록 (필터/정렬)

### Detail2 탭 — 가격 & 동시접속자 분석
- 카드 4개 (현재 가격, 최고 할인율, 최고/최저 동시접속자)
- 시간대별 동시접속자 히트맵 (3시간 bin × 날짜, 할인일 🏷 표시)

### 공통
- 패치노트 클릭 시 전 탭이 해당 날짜 기준 기간으로 연동 (anchor 기능)
- AI 코멘트 팝오버 🤖 (3 섹션 9 항목, Gemini 2.5 Flash 자연어 요약, 통계 검정 기반)

## 데이터 수집 전략

### 수집 흐름

```
Python 스크립트 → Steam API 호출 → 데이터 수신 → BigQuery에 직접 INSERT
(RDS에 INSERT INTO 하듯이, BigQuery client.insert_rows_json()으로 직접 적재)
```

### 수집 대상 & 주기

| 테이블 | 수집 시점 | 방식 |
|---|---|---|
| **games** | 초기 1회 + 매일 | 초기: 소유자순 상위 500개 수집. 매일: 트렌딩 50개 중 DB에 없는 신규만 추가 |
| **price_history** | 매일 | DB 전체 게임 가격 스냅샷 append |
| **news** | 매일 | 전날 발행된 뉴스만 append (KST 기준) |
| **reviews** | 매일 | 전날 작성된 리뷰만 append (KST 기준) |
| **review_summary** | 매일 | 리뷰 수집 시 같이 갱신 (최신 통계, TRUNCATE 후 재삽입) |
| **player_counts** | 3시간마다 | 현재 동시접속자 수 append (하루 8회) |

### 게임 목록 관리

- **초기 수집**: SteamSpy `all` API → 소유자 수 기준 상위 500개
- **매일 갱신**: Steam Korea 스토어 트렌딩 상위 50개 확인 → DB에 없는 게임만 추가
- 시간이 지날수록 DB에 게임이 자연스럽게 축적되는 구조

### 일별 증분 수집 로직

- **reviews**: `filter=recent` + `timestamp_created`로 KST 전날 리뷰만 필터링
- **news**: `date` 필드(unix timestamp)로 KST 전날 뉴스만 필터링
- **price_history**: `snapshot_date`로 하루 1회 스냅샷 (할인 시작/종료 감지)
- **review_summary**: 매일 TRUNCATE 후 최신 통계 재삽입 (게임당 1행)

## 데이터 수집 대상 상세

### 1. 게임 기본 정보
- 이름, 가격, 할인율, 장르, 태그, 출시일, 개발사, 퍼블리셔
- **소스**: Steam Store API (`appdetails`)

### 2. 유저 리뷰
- 리뷰 텍스트, 추천/비추천, 플레이 시간, 작성일
- **소스**: Steam Reviews API (`appreviews`) — API 키 불필요 (공개)
- **활용**: 키워드 분석, 감성 분석, AI 요약

### 3. 가격/할인 이력
- Steam API에서 공식 제공하지 않음 → **매일 스냅샷으로 직접 구축**
- 신규 게임은 IsThereAnyDeal API로 과거 이력 백필
- 할인 시작/종료 시점, 할인 기간, 할인 빈도 분석 가능

### 4. 동시접속자 수
- **소스**: Steam Web API (`GetNumberOfCurrentPlayers`)
- 3시간마다 수집 → 시간대별 히트맵 + 일별 추이 분석 가능

### 5. 업데이트/패치노트
- **소스**: Steam News API (`GetNewsForApp`)
- **활용**: 패치 날짜를 기준일(anchor)로 설정 → 전후 리뷰/동접자 변화 분석

## AI 기능 (구현 완료)

대시보드 좌측 상단 **🤖 AI 코멘트** 팝오버에서 3개 섹션 9개 항목을 자연어로 제공.

### 설계 컨셉

- **그로스 마케팅 lens 단일** — 통계 검증이 아니라 의사결정 보조 도구
- 분석은 Python(BQ 쿼리 + 통계)이 수행, Gemini는 결과 dict를 받아 **친근한 자연어로 표현만 변환**
- 환각 방지: "데이터에 없는 사실 절대 추측·생성 금지" 프롬프트 명시
- 톤: p-value/표준편차 같은 통계 용어는 금지하되, "통계적 검정 결과 유의미한 변화"·"평소 변동 범위 안" 같은 평이한 표현은 허용
- 출력 패턴: **구체 숫자 → 통계 검정 결론 → "따라서 ~" 액션 시그널**

### 모듈 구조

```
analyses/
├── sentiment_summary.py     # 섹션 ① 통합 — TF-IDF/추세 결합
├── tfidf_keywords.py        # 섹션 ① 항목 2 — 차별화 키워드 (전체/장르)
├── segment_signals.py       # 섹션 ② — 코호트 + 언어 갭
├── event_signals.py         # 섹션 ③ — 패치/세일/키워드/동접 이상치
└── gemini_comment.py        # 3 섹션 프롬프트 + Gemini 호출 + fallback 템플릿
```

각 모듈은 `client, app_id`만 받아 dict 반환 → 테스트 가능, 캐싱 친화적.

### 9개 분석 항목 — 데이터/통계 방법

#### 섹션 ① 감성·키워드 요약 (현황)
> "총 53만 건이 넘는 리뷰 중 긍정률이 94.1%, 멀티플레이가 긍정 의견의 38.9%를 차지하며 '건설'·'건축'이 이 게임만의 차별점으로 두드러집니다. 부정 의견 중 '반복성'이 80%로 가장 큰 비중을 차지해 운영 우선순위 검토가 필요합니다."

| # | 항목 | 데이터 | 방법 |
|---|------|--------|------|
| 1 | Top 긍정/부정 카테고리 | `category_analysis` | polarity별 합산 → share_pct 계산 → Top 3 |
| 2 | TF-IDF 차별화 키워드 | `keyword_analysis` | TF=ratio, IDF=ln(N/df). df≥2, count≥3 필터. **전체 + 같은 장르 (genres prefix 2단어 일치)** 두 코퍼스 비교, 장르 코퍼스<10이면 장르 비교 생략 |
| 3 | 긍정률 4주 추세 | `review_summary` (일별 누적) | 주별 마지막 스냅샷 → 인접 주 delta로 "그 주에 새로 들어온 리뷰의 긍정률" 산출. 최근주 vs 직전 3주 가중평균 비교 |

#### 섹션 ② 유저 세그먼트 시그널 (구조)
> "헤비유저(50시간 이상)의 만족도가 67.3%로 평균 대비 낮아 long-term 만족도 개선이 필요합니다. 중국어 간체 사용자는 60.3%로 최저 — 현지화 우선순위 시그널입니다."

| # | 항목 | 데이터 | 방법 |
|---|------|--------|------|
| 4 | 플레이타임 코호트 | `reviews.playtime_at_review` | 0-10h / 10-50h / 50h+ 버킷, n≥30 필터, 버킷별 추천률 + 전체 평균 |
| 5 | 언어별 추천률 갭 | `reviews.language` | Top 5 언어 (n≥30), 최고/최저/갭_pp 계산 |

#### 섹션 ③ 변화 알림 (이벤트)
> "60% 할인 세일 동안 동접이 직전 대비 22.1% 늘었지만, 통계적 검정 결과 평소 변동 범위 안이라 세일 효과로 단정하긴 어렵습니다. 따라서 가격 인하 자체보다 다른 요인 영향일 가능성도 봐야 합니다."

| # | 항목 | 데이터 | 방법 |
|---|------|--------|------|
| 6 | 패치 임팩트 | `news` + `reviews` + `player_counts` | 최근 60일 내 뉴스 1건 = 이벤트. ±7일 윈도. **추천률은 두 비율 z-test (pooled SE), 동접은 비-패치 baseline z-score** |
| 7 | 세일 효과 | `price_history` + `player_counts` | 최근 연속 할인 윈도(`discount_percent ≥ 10%`). **세일 중 일별 평균 vs 비-세일 baseline z-score** |
| 8 | 부정 키워드 급증 | `keyword_analysis` (시계열) | 최근 1주 vs 직전 4주 평균. **급증=ratio≥2배**, **신규=직전엔 0회** |
| 9 | 동접 z-score 이상치 | `player_counts` + `news` + `price_history` | 30일 baseline μ/σ → 최근 7일 z 계산. **\|z\|≥2 일자**에 ±2일 뉴스/할인 cross-ref |

### 통계적 유의성 적용

단순 % 임계값을 쓰면 게임마다 변동성이 달라 거짓 시그널이 나옴. 각 게임의 자체 변동성을 baseline으로 쓰는 z 기반으로 통일.

- **두 비율 z-test (pooled SE)** — 항목 6의 추천률 변화. `_two_proportion_z(pos1,n1,pos2,n2)` → `(p2-p1) / sqrt(p_pool*(1-p_pool)*(1/n1+1/n2))`. 표본이 작으면 SE가 커져 자동으로 z가 작아지므로 별도 필터 불요.
- **Baseline z-score (1-sample)** — 항목 6/7/9의 동접 변화. baseline = 60일 중 이벤트 ±7일 제외한 일별 동접 평균. `z = (관심기간 평균 - baseline μ) / baseline σ`.
- **5단계 카테고리화** — 양쪽 z 모두 |z|=2 (≈95% CI), |z|=1 (약함) 기준:
  - 추천률: 뚜렷한_상승 / 약한_상승 / 유의미한_변화_없음 / 약한_하락 / 뚜렷한_하락
  - 동접: 뚜렷한_증가 / 약한_증가 / 평소_변동_범위 / 약한_감소 / 뚜렷한_감소

**이 접근의 가치 — 실제 사례**
- 단순 % 비교로는 "할인 후 동접 +22%"가 "효과적"으로 잡힘 → 한 게임의 평소 σ가 7,000명이면 +22%는 평소 변동 범위 안
- 단순 % 비교로는 "패치 후 -5.4%"가 "패치 부정적"으로 보임 → baseline 대비로는 약한 증가 (직전 7일이 일시적으로 높았던 것뿐)

### Gemini 프롬프트 설계

3개 섹션 각각 전용 프롬프트 (`SECTION1_PROMPT` / `SECTION2_PROMPT` / `SECTION3_PROMPT`). 공통 원칙:

```
1. 데이터 dict 구성을 명시 (어떤 키가 무엇을 의미하는지)
2. 톤 가이드 (그로스 마케터 보고 톤, 평이한 한국어)
3. 항목별 서술 패턴 (예: 부정 카테고리는 share_pct 인용, TF-IDF 키워드는 전체/장르 구분)
4. 결측 처리 룰 (None이면 통째로 언급 금지, "데이터 없음" 메타 코멘트도 금지)
5. 표현 화이트리스트/블랙리스트
   - 허용: "통계적 검정 결과 유의미한 변화", "평소 변동 범위 안"
   - 금지: "z-score", "표준편차", "p-value", "ROI", "신뢰구간"
```

특히 섹션 ③은 **"구체 숫자 → 통계 결론 → 따라서 ~ 액션 시그널"** 순서를 강제해서 출력이 일관됨.

### 기술 구현 흐름

```
1. Python 분석 (BQ 쿼리 + 통계 헬퍼)
   → 분석 결과 dict 생성 (TF-IDF, z-score, 유의성 카테고리 등)

2. Gemini 2.5 Flash (Vertex AI) 호출
   → SYSTEM_PROMPT + 분석 dict → 자연어 코멘트
   → thinking_config=ThinkingConfig(thinking_budget=0) 으로 빠른 응답

3. Streamlit dashboard 후처리
   → _highlight_text(): 따옴표 안 한글 키워드 + 숫자/% 패턴을 골드 색 강조
   → \n → <br> 변환

4. Fallback 템플릿 (API 장애 시) — 각 섹션마다 f-string 평문 narrative 제공
```

### UX 디자인

- **단일 트리거 버튼** — popover 제목 옆 우측 정렬. 클릭 시 `ThreadPoolExecutor(max_workers=3)`로 3 섹션 **병렬 실행** (직렬 ~15초 → 병렬 ~5-8초)
- **이중 캐싱**:
  - `@st.cache_data(ttl=3600)` — 프로세스 레벨 (재실행 시 BQ/Gemini 재호출 안 함)
  - `st.session_state[f"ai_*_{app_id}"]` — 브라우저 세션 레벨 (popover 열고 닫아도 결과 유지)
- **숫자·키워드 강조**:
  - 정규식으로 `12.9%p`, `53만 건`, `1,234,567명`, `2배` 매칭 → 골드 색 굵게
  - `'재미'`, `'밸런스'`, `'중국어 간체'` 같은 따옴표 안 한글 키워드 매칭 (날짜 단위 일/월/년은 제외)
- **카드 스타일** — 다크 배경 `#1B2838`, 좌측 Steam 블루 액센트 바, `word-break:keep-all`로 한글 단어 안 깨짐
- **상태 표시** — 미실행 시 placeholder 안내, 실행 후 카드 표시. 모두 실행되면 버튼 라벨 "▶ 분석 실행" → "🔄 재실행"

### 모델 선택 — Gemini 2.5 Flash (Vertex AI)

- GCP 인프라에 통합 — BQ/Cloud Functions와 같은 서비스 계정 사용 (별도 API 키 관리 불필요)
- 비용 청구 통합
- 코멘트 수준 출력엔 Claude/Gemini 품질 차이 미미, 비용은 Gemini Flash가 유리

### 비용 추정

게임 1개 분석 시 — BQ 쿼리 ~10회 + Gemini 호출 3회. 캐시 적중 시 0회.

| Gemini 2.5 Flash (입력) | $0.075/1M tokens | 프롬프트 ~1k tokens × 3 → ~$0.0002 |
| Gemini 2.5 Flash (출력) | $0.30/1M tokens | 응답 ~300 tokens × 3 → ~$0.0003 |
| **합계** | | **~$0.001 / 게임 / 1회** |

캐시 (1시간 TTL) 적중률 50% 가정 시 사용자당 비용은 절반.

## 기술 스택 & 아키텍처

### 인프라: GCP (Google Cloud Platform)

| 컴포넌트 | 기술 | 역할 |
|---|---|---|

| 프로토타입 대시보드 | Streamlit | 현재 서비스 중인 분석 대시보드 |
| 백엔드 (예정) | FastAPI | BigQuery 읽어서 API 서빙 |
| 프론트엔드 (예정) | React + Vercel | 소비자용 페이지 + 개발자용 대시보드 |
| AI (예정) | Gemini 2.5 Flash (Vertex AI) | 분석 결과 → 자연어 코멘트 변환 |

### 데이터 파이프라인 흐름

```
Cloud Scheduler (매일 트리거)
        ↓
Cloud Functions (Python / functions-framework)
        ↓  Steam API / ITAD API 호출
        ↓  client.insert_rows_json()으로 직접 적재
BigQuery steam_data (원본 데이터 저장)
        ↓  SQL Views (타임스탬프 변환 + 게임명 JOIN)
Streamlit Dashboard (프로토타입)
```

- Cloud Functions 5종 작성 (collect_daily, process_new_games, translate_daily, analyze_daily, collect_player_counts)
- Cloud Scheduler 자동화 배포 완료

### Phase 2: 분석 레이어 — 완료

- 리뷰 한국어 번역 파이프라인 (Deep Translator, 일 최대 8,000건)
- Kiwi 기반 형태소 분석 + 감성 분류 (긍/부정)
- 카테고리별 분류 (24개 서브카테고리) + 키워드 빈도 추출 TOP 30
- BigQuery SQL 집계 쿼리 및 Views 구성

### Phase 3: Streamlit 대시보드 — 완료

3탭 구조 (Main / Detail / Detail2), Steam 스타일 다크 테마

**Main 탭**
- 게임 검색 + 기본 정보 (이미지, 장르, 메타크리틱)
- 카테고리 배지 전체 표시
- 리뷰 요약 카드 (총 리뷰, 긍정률 게이지, 통합평가)
- 카테고리 레이더차트 (긍정 7축 / 부정 7축)
- 패치노트/뉴스 목록 + 기준일(anchor) 설정 기능
- 현재 가격 + 평균 동시접속자 카드
- 동시접속자 일별 라인차트
- 할인 내역 테이블

**Detail 탭**
- KPI 카드 (총 리뷰, 통합평가, 평균 플레이타임, 긍정 키워드 수)
- 일별 플레이타임 추이 (추천/비추천 분리 라인차트)
- 일별 리뷰수 스택바차트 (긍정 비율 레이블)
- 언어 비중 파이차트 (추천/비추천 분리)
- 키워드 빈도 표 TOP 20 (추천/비추천 분리)
- 리뷰 목록 (추천 여부·언어·번역/원문·정렬 필터)

**Detail2 탭**
- 카드 4개 (현재 가격, 최고 할인율, 최고/최저 동시접속자)
- 시간대별 동시접속자 히트맵 (3시간 bin × 날짜, 할인일 🏷 표시)
- anchor 연동 (패치일 기준 14일 창)

**공통**
- AI 코멘트 팝오버 🤖 (3 섹션 9 항목 통계 기반 자연어 요약 — Phase 4 참고)
- anchor 기능: 패치노트 클릭 시 전 탭이 해당 날짜 기준 기간으로 연동

### Phase 4: AI 기능 백엔드 — 완료

그로스 마케팅 lens 단일, 분석은 Python(BQ 쿼리 + 통계), 자연어 변환만 Gemini 2.5 Flash (Vertex AI). 9개 항목 전부 구현.

- **Phase 4-1 — 빠른 승부 (단순 SQL)**
  - ① Top 카테고리/키워드 (share_pct 포함), 긍정률 4주 추세 (review_summary append 모드 전환)
  - ② 플레이타임 코호트 추천률, 언어별 추천률 갭
  - Gemini 연결, Streamlit popover 단일 트리거 + 병렬 실행 + 세션 캐시
- **Phase 4-2 — 차별화/변화 감지**
  - ① TF-IDF 차별화 키워드 (전체 / 같은 장르 두 코퍼스)
  - ③ 부정 키워드 급증 (이번주 vs 4주 평균), 동접 z-score 이상치 + cross-ref
- **Phase 4-3 — 인과 임팩트**
  - ③ 패치 임팩트 — 추천률 두 비율 z-test, 동접 baseline z-score
  - ③ 세일 효과 — 비-세일 baseline z-score
- **Phase 4-4 — 가독성/UX 다듬기**
  - 통계적 톤 완화 (구체 숫자 → "통계적 검정 결과 유의미한..." → "따라서 ~")
  - popover 폭 확장(520~640px), 카드 다크 테마, 좌측 액센트 바
  - 숫자/키워드 정규식 강조 (골드 색)

### Phase 5: 배포 — 진행 중 (포트폴리오 공개 모드)

Streamlit Community Cloud 무료 배포. 단 AI 코멘트는 Gemini API 비용이 호출당 ~$0.001 발생 → 공개 시 비용 통제 필요.

배포 옵션 (구현 결정 단계):
1. **Pre-compute 캐시 모드 (추천)** — Top 30개 인기 게임에 대해 매일 1회 AI 코멘트 미리 계산해서 BQ에 저장. 사용자는 캐시된 결과만 봄. 임의 게임 검색 시 AI 버튼 비활성화 또는 "데모 영상 참고"
2. **AI 버튼 hide 모드** — 공개 버전엔 AI 코멘트 popover 자체 비활성화. 데모는 README의 스크린샷/영상으로
3. **인증 게이트** — 본인만 접근 가능하게 Cloudflare Access나 Streamlit auth로 보호. 포트폴리오 검토자에게 별도 링크 공유

GCP 인증은 `credentials.json` 대신 Streamlit Secrets로 전환.
